# Claude Code 開発フローガイド

ブラウザ版AIでのコーディングに慣れているエンジニアが、はじめてClaude Codeを使うための実践ガイドです。**ターミナル（CLI）版** と **VSCode拡張機能版** の両方に対応しています。

---

## ブラウザAIとの違い

| 項目 | ブラウザ版AI（Gemini等） | Claude Code |
| :--- | :--- | :--- |
| 実行環境 | ブラウザ | ターミナル / VSCode |
| ファイル操作 | できない（コピペが必要） | **直接読み書き可能** |
| コマンド実行 | できない | **シェルコマンドを実行可能** |
| コンテキスト | 手動でコピペ | **カレントディレクトリを自動認識** |
| 費用 | 無料または月額固定 | **APIトークンを消費** |

最大の違いは「Claudeが自分でファイルを変更・コマンドを実行する」点です。
操作のたびに確認を求めてきますが、**何に対してOKするかを意識する**ことが重要です。

---

## 起動方法

### ターミナル版

初回のみインストールが必要です。

```bash
curl -fsSL https://claude.ai/install.sh | bash
```

インストール後、プロジェクトのディレクトリで起動します。

```bash
cd /path/to/project
claude
```

### VSCode拡張機能版

VSCode のサイドバーまたはコマンドパレット（`Cmd+Shift+P` → "Claude Code"）からチャットパネルを開きます。

## 最初にやること

タスク開始時に `@claude-dev-flow.md` を参照させてください。開発フローのルールをClaudeに読み込ませることで、計画・実装・まとめの3ステップを正しく進めてくれます。

```
@claude-dev-flow.md ○○を実装したい。
```

## 開発の3ステップ・ループ

### ステップ1：計画（Plan）— 仕様の合意

いきなり書かせず、まず「何をどう実装するか」を確認します。

**やること:** 目的の伝達 → 既存コードの調査 → 仕様の壁打ち → 詳細設計の合意

**プロンプト例:**
> 「src/components 以下のボタンコンポーネントをリファクタしたい。現在の実装の課題を挙げて、修正案を提案して。不明点があれば質問して。」

**ファイルの指定方法:**

共通:
- チャット欄で `@` と入力するとファイル選択メニューが開く

ターミナル版のみ:
- パイプでファイルの内容を渡すことができる
  ```bash
  cat src/main.php | claude -p "このコードをレビューして"
  ```

**詳細設計の合意:**

仕様が固まったら、実装に入る前にClaudeに詳細設計をまとめさせ、開発者と合意を取ります。

> 「仕様を踏まえて、以下の観点で詳細設計をまとめてください。
> - 変更・追加するファイルの一覧
> - 各ファイルへの変更内容（関数・クラス・ロジックの変更点）
> - 既存の実装で流用できる箇所
> - 懸念点やリスク
> 内容を確認してOKなら実装に進みます。」

---

### ステップ2：実行（Do）— 合意後に開始

方針に納得したら、明確にGOサインを出します。

**プロンプト例:**
> 「その方針でOKです。実装してください。終わったら `npm test` で動作確認まで完了させて。エラーが出たら自己修正して。」

> ※ テストコマンドはプロジェクトによって異なります（`go test ./...`、`pytest` 等）。テストが存在しない場合はその旨をClaudeに伝えてください。

**権限確認（y/n）について:**

実装中、Claudeは以下の操作で確認を求めます。内容を確認してから `y` を押してください。

| 確認の種類 | 内容 |
| :--- | :--- |
| ファイル書き込み | 変更するファイルと変更内容 |
| コマンド実行 | `npm test` や `git` などのコマンド |

> **内容を確認せずに `y` を押し続けるのは避けてください。**

**レビュー → 修正ループ:**

テストがパスしたらClaudeが実装完了を報告し、開発者のレビューを受けます。指摘があれば修正して再レビューを依頼し、問題がなくなるまで繰り返します。

```
① Claudeが実装完了を報告 → 開発者にレビューを依頼
② 開発者がコードをレビュー
   ├── 指摘あり → ③へ
   └── 問題なし → 完了
③ Claudeが修正 → テスト実行 → ①へ戻る
```

修正を依頼するプロンプト例:
> 「〇〇のロジックが仕様と異なります。△△のように修正してください。」

---

### ステップ3：まとめ（Wrap-up）— 仕上げ

**プロンプト例:**
> 「今回の変更内容を箇条書きでまとめて。PRのタイトルと概要を作成して。」

## コマンド一覧

### 共通

| コマンド | 内容 |
| :--- | :--- |
| `/undo` | 直前のファイル操作を取り消す |
| `/compact` | 会話が長くなり挙動が不安定になったときに実行（コンテキストリセット） |
| `/cost` | 現在のセッションの利用料金を確認 |
| `@filename` | 特定のファイルを読み込ませる（精度向上） |

### ターミナル版

| コマンド | 内容 |
| :--- | :--- |
| `Ctrl+C` | Claudeの動作を強制停止 |
| `claude -c` | 前回の会話を続行 |
| `claude -p "質問"` | ワンショット実行（回答後に自動終了） |

### VSCode拡張機能版

| コマンド | 内容 |
| :--- | :--- |
| `Esc` | Claudeの動作を強制停止 |

## トラブル時の対処

### Claudeが同じエラーを繰り返す場合

作業を止め、状況を整理してから改めて指示を出し直します。

> 「一度止めます。現在の状況を整理して、次のアクションを提案してください。」

### ファイルの変更を元に戻したい場合

| 状況 | 操作 |
| :--- | :--- |
| 直前の1操作だけ戻したい | `/undo` |
| 複数の変更をまとめて戻したい | `git restore .` |

> `git restore .` は、作業前にブランチを切っていることが前提です。

### 作業前のブランチ確認

Claude Codeはファイルを直接変更します。**作業前に必ず専用ブランチを切ってから始めてください。** 万一の場合も `git` で安全に元に戻せます。

## 注意事項

1. **機密情報の扱いに注意:** 本番環境のシークレットキーなどはチャットに含めないでください。
2. **スコープを明確に:** 「このファイルの、この関数だけ変更して」のように範囲を絞ると精度が上がります。
3. **トークン節約:** 巨大なディレクトリを丸ごと読み込ませるより、必要なファイルだけを `@` で指定する方が高速で安価です。
